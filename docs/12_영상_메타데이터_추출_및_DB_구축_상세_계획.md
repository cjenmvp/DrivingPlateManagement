# 영상 메타데이터 추출 및 DB 구축 상세 계획 (v1.0)

본 문서는 파일명이 비표준화되어 맥락 파악이 어려운 `X:\01_DrivingPlate_Original` (25TB) 자산을 정화하고, 기술적 근거에 기반한 자산 관리 시스템(SQLite DB)을 구축하기 위한 실행 계획서입니다.

---

## 🛠️ 1단계: Pilot Test (ffprobe 기술 검증)
대규모 스캔 전, 시스템 부하와 추출 데이터의 정확도를 검증합니다.
- **대상**: 비표준 폴더 3개 및 표준 폴더 7개 (총 10개 샘플)
- **도구**: `ffprobe` (FFmpeg)
- **추출 항목**: 
  - `Duration` (정확한 영상 길이 - 초 단위)
  - `Width/Height` (해상도 - 4K/FHD 판별)
  - `r_frame_rate` (정확한 FPS - 24p/60p 판별)
  - `Codec_name` (NotchLC/H264/ProRes 판별)
  - `Bitrate` (화질 수준 판별)
- **출력**: `temp_metadata.json`

## 🏗️ 2단계: SQLite DB 스키마 설계
추출된 정보를 체계적으로 담을 그릇을 설계합니다. (파일명이 바뀌어도 데이터는 유지되도록 설계)
- **Table: `folders`**: 폴더 경로, 추측 날짜, 장소명, 상태 태그.
- **Table: `video_files`**: 파일명, 용량(Size), 확장자, 폴더 ID.
- **Table: `technical_metadata`**: 해상도, FPS, 길이, 코덱, 비트레이트 (File ID와 1:1 매칭).
- **Table: `sync_status`**: 마스터(Z:)와의 매핑 여부, 중복 여부 체크.

## 🔄 3단계: Batch Processing (전수 조사)
메모리 누수를 방지하며 25TB 전체 데이터를 순차적으로 스캔합니다.
- **방식**: 10개 폴더 단위로 묶어서 `ffprobe` 실행 → 결과 DB 입력 → 메모리 해제 (Chunking).
- **안전 장치**: 오류 발생 시 해당 폴더를 스킵하고 로그를 남겨 시스템 중단을 방지.

## 🧠 4단계: 맥락 재구성 및 정화 (Context Reconstruction)
DB에 쌓인 기술 정보를 바탕으로 엉망인 파일들을 '이해'합니다.
1. **중복 식별**: 파일명은 달라도 `Size`와 `Duration`이 100% 일치하면 **중복 파일**로 간주하여 격리.
2. **이름 복구**: `Duration`과 `FPS` 정보를 마스터(Z:)의 NotchLC와 대조하여, 깨진 이름의 원본 폴더명을 **표준 명칭(`YYMMDD_...`)으로 자동 복구**.
3. **용량 최적화**: 기술 정보가 없는 찌꺼기 파일(Tmp, log 등)을 식별하여 일괄 삭제 또는 이동.
