# 대규모 영상 데이터 DB 및 메타데이터 도입 계획 (2026-03-16)

현재 30TB 이상의 방대한 드라이빙 플레이트 원본 데이터를 파일 시스템 기반으로 직접 스캔/관리할 경우, 메모리 누수 및 시스템 크래시가 발생할 위험이 큽니다. 이를 해결하고 엔터프라이즈급 자산 관리를 위해 데이터베이스(DB)와 메타데이터 추출 자동화를 도입하는 계획입니다.

---

## 🚨 1. 현재의 문제점 (Pain Points)
- **메모리 부하**: `Get-ChildItem` 등 파일 탐색 명령어로 수만 개의 대용량 영상 파일(.mov, .nclc)을 읽어 들일 때 메모리 스파이크 발생.
- **검색 속도 저하**: 특정 조건(예: "야간, 60fps, 360 리플렉션")의 파일을 찾기 위해 매번 디스크 전체를 읽어야 함.
- **데이터 파편화**: 엑셀 수동 기록과 실제 파일 상태 간의 싱크가 어긋날 위험 높음.

---

## 🏗️ 2. 시스템 아키텍처 전환 전략 (DB 도입)

### A. 가벼운 로컬 DB (SQLite) 도입
무거운 엔터프라이즈 DB 대신, 빠르고 가벼운 **SQLite**를 도입하여 운영 문서 경로(`X:\00_DrivingPlate_Management_System\library.db`)에 저장합니다.
- **기능**: 모든 폴더와 파일의 경로, 용량, 태그를 DB 테이블로 관리.
- **효과**: 파일 검색 시 디스크를 읽지 않고 DB에 쿼리문(SQL)만 날리므로 메모리 부하가 `0`에 가깝습니다.

### B. 메타데이터 자동 추출 (FFprobe 활용)
영상을 플레이하지 않고 헤더(Header) 정보만 아주 빠르게 읽어내는 `ffprobe` (FFmpeg 도구)를 활용합니다.
- **추출 항목**: 정확한 영상 길이(Run Time), 해상도, 코덱(NotchLC/H264), 프레임 레이트(FPS).
- **작업 방식**: 신규 파일이 입고될 때만 스크립트가 돌아가 정보를 추출하고 DB에 `INSERT` 합니다.

## ⚙️ 3. 메모리 누수 방지(Safe Processing) 워크플로우

과거 데이터를 최초로 DB에 넣을 때(Migration) 메모리 문제를 막기 위한 로직입니다.

1. **Chunking (분할 처리)**: 한 번에 폴더 10개씩만 메모리에 올린 후 DB에 기록하고 메모리를 즉시 비움(Garbage Collection).
2. **Watch Folder (감시 폴더)**: 전체 스캔을 없애고, 특정 폴더(`02_Work_In_Progress`)에 파일이 복사/삭제되는 이벤트(Event)만 감지하여 DB를 업데이트.
3. **MAM (Media Asset Management) 소프트웨어 연동 고려**: 장기적으로는 DB 구축을 넘어 Kyno, 달스톤(Dalet) 등 전용 미디어 관리 툴 도입을 대비하여 메타데이터를 범용 포맷(JSON/XML)으로도 출력.
